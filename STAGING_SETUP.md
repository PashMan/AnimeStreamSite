# Настройка Staging-сервера (Тестового окружения)

Так как вы используете (или планируете использовать) **Cloudflare Pages**, настройка тестового сервера (Staging) делается очень легко, потому что Cloudflare поддерживает это "из коробки".

## Шаг 1: Создание тестовой базы данных
Самое главное правило Staging — **никогда не использовать основную базу данных для тестов**.
1. Зайдите в [Supabase](https://supabase.com/).
2. Создайте **новый проект** (например, `kamianime-staging`).
3. Скопируйте структуру вашей основной базы (таблицы, политики) в новую базу. Вы можете использовать SQL-скрипты из вашего проекта (например, `supabase_schema.sql`).
4. Получите `URL` и `anon key` от нового тестового проекта.

## Шаг 2: Настройка переменных окружения в Cloudflare
1. Зайдите в настройки вашего проекта в **Cloudflare Pages**.
2. Перейдите в **Settings -> Environment variables**.
3. Вы увидите две вкладки: **Production** и **Preview**.
   * **Production:** Здесь должны быть ключи от вашей *основной* базы Supabase.
   * **Preview (Это и есть ваш Staging):** Добавьте сюда переменные `VITE_SUPABASE_URL` и `VITE_SUPABASE_ANON_KEY`, но вставьте значения от вашей *тестовой* базы (из Шага 1).

## Шаг 3: Работа с Git (Ветки)
Вам нужно создать отдельную ветку для тестирования.
1. В вашем репозитории создайте ветку `staging` (или `dev`).
2. Когда вы делаете новые фичи, отправляйте (push) их в ветку `staging`.

## Шаг 4: Как происходит тестирование
1. Вы пушите код в ветку `staging`.
2. Cloudflare Pages автоматически замечает это и собирает проект.
3. Он выдаст вам уникальную ссылку (например, `https://staging.kamianime.pages.dev`).
4. Этот сайт будет работать с вашей **тестовой базой данных** (благодаря настройкам из Шага 2).
5. Вы тестируете сайт. Если всё хорошо — вы делаете Pull Request и сливаете (merge) ветку `staging` в ветку `main`.
6. Cloudflare собирает ветку `main` и обновляет ваш **основной сайт**, который работает с **основной базой**.

## Локальная разработка
Для удобства локальной разработки я создал файлы:
* `.env.staging` (шаблон для тестовых ключей)
* `.env.production` (шаблон для боевых ключей)

Если вы хотите запустить проект локально с тестовой базой, вы можете переименовать `.env.staging` в `.env` (или скопировать значения).
